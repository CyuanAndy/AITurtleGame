## 2. 项目概述

### 2.1 产品与技术边界

使用 **React + TypeScript + Vite** 与 **Tailwind CSS** 开发的 **AI 海龟汤** Web 应用：玩家在大厅选题，在游戏页与 AI 主持人对话，仅获得「是 / 不是 / 无关」类回答，可提交真相、查看汤底或结束游戏。后端（Node.js + Express 等）负责会话、`truth` / `core_conditions`、AI 代理与限流；前端不持有密钥与完整汤底。

---

## 3. 开发规范

### 3.1 规范清单

- 使用 **TypeScript**，避免滥用 `any`，公共类型集中管理（如 `src/types`）。
- 使用 **函数式组件 + Hooks**；会话相关状态可用 **Context**（见 `TECH_DESIGN.md` §2.3、§3.1）。
- 使用 **Tailwind CSS** 编写样式，避免全局随意堆叠；与 PRD 调性一致（见第 5 章）。
- **组件**应可复用、职责单一；**注释**用于解释非显而易见的设计（如会话恢复、防刷逻辑），避免为显而易见的代码写冗长注释。
- **路由**使用 React Router；页面与 `TECH_DESIGN.md` 中的大厅 / 游戏 / 汤底划分对齐。
- **安全**：`truth`、`core_conditions`、**所有 API Key 仅服务端**；前端仅用环境变量配置 **前端** 可知的项（如 `VITE_API_BASE_URL`），**禁止**在仓库或客户端代码中硬编码私钥。
- **与 PRD/技术设计对齐**：滑动窗口、JSON 模式判题、提交真相次数与重新判定次数等以后端与 `TECH_DESIGN.md` 为准，前端不得伪造权威对话历史。

---

## 4. 代码风格

### 4.1 命名与类型约定

- **组件名**：PascalCase（如 `GameCard.tsx`）。
- **函数 / 变量名**：camelCase。
- **常量**：`UPPER_SNAKE_CASE`。
- **类型**（接口 / type alias）：以 **`T` 开头**（如 `TStory`、`TMessage`、`TSession`），与项目约定一致。
- **文件与目录**：与 `TECH_DESIGN.md` §3.1 建议结构保持一致；新增文件时沿用现有命名习惯。

---

## 5. 设计要求

### 5.1 视觉与交互

- **整体风格**：神秘、悬疑、沉静；**深蓝**背景（如 `bg-slate-900` 或 PRD 推荐的深蓝区间）。
- **强调色**：金色点缀（如 `text-amber-400`）用于按钮与关键信息，避免刺眼荧光。
- **圆角**：`rounded-lg`（或同层级统一圆角）。
- **阴影**：`shadow-lg` 用于卡片、重要容器，保持克制。
- **布局**：移动优先，响应式网格；游戏页类 IM 布局，底栏不遮挡最后一条消息（见 PRD §8）。
- **无障碍**：关键按钮尽量可聚焦；重要操作有 `aria-label`（后续迭代可补全）。

---

## 6. 注意事项

### 6.1 工程原则

- **保持简洁**，避免过度抽象；改动范围聚焦需求，**不做无关大重构**。
- **优先 MVP 核心闭环**：大厅 → 游戏对话 → 汤底 / 结束；再迭代体验细节。
- **移动端适配**为必选项；关键路径在窄屏下可完整操作。
- **API Key 与环境变量**：服务端密钥仅通过部署环境注入；前端 README 中可说明需填写的变量名，**禁止**把真实密钥写入代码或提交到版本库。
- **错误与加载**：网络错误、AI 等待需有明确 Loading 与重试提示（对齐 PRD G-8、G-9）。

---

## 7. 测试要求

### 7.1 手动测试要点

- **每完成一个功能点** 在本地手动走通相关路径。
- **布局**：在常见宽度下检查（如窄屏手机、平板、桌面），确保列表、聊天区、底部操作区不重叠、不截断关键文案。
- **会话**：刷新页面后，若产品要求恢复会话，验证 `sessionId` 与 `GET /api/sessions/:id` 行为符合 `TECH_DESIGN.md` §2.3；无效会话应回大厅并清理本地状态。
- **AI 行为**：联调时检查回答是否仅「是 / 不是 / 无关」、提交真相与重新判定是否受次数约束；后端 JSON 解析失败时是否有降级或提示（见 `TECH_DESIGN.md` §6、§7）。
