## 1. 产品概述

### 1.1 一句话描述

一个 **AI 驱动的海龟汤游戏网站**：玩家与 **AI 主持人** 通过对话提问，在仅获得「是 / 否 / 无关」三类回答的前提下，推理出故事背后的 **汤底（真相）**。

### 1.2 产品定位

| 维度 | 说明 |
|------|------|
| 品类 | 情境推理 / 文字解谜 / 轻社交娱乐 |
| 核心价值 | 低门槛即时推理、AI 标准化主持、可持续内容（题库 + 未来生成） |
| 差异化方向（MVP 后） | 题目质量与 **回答一致性**（避免前后矛盾）、结构化复盘 |

### 1.3 背景与动机

- **线下海龟汤** 依赖主持人能力与组局成本，线上化可降低门槛。
- **大模型** 可承担「读题—判问—只答三类」的主持人角色，但需 **强规则约束**，否则体验崩坏。
- 市面已有类似形态，MVP 需先跑通 **大厅 → 单局 → 汤底** 闭环，再迭代账号、排行与多人。

### 1.4 范围说明（本文档）

- **包含**：MVP 功能规格、页面与交互、AI 行为约束、非功能需求、验收标准、技术栈建议、里程碑。
- **不包含**：具体视觉稿（仅风格与组件级要求）、具体营销文案定稿、商务与定价（可后续单独立项）。

## 2. 目标与成功指标

### 2.1 产品目标（MVP）

1. 用户可在 **3 步内** 开始一局：进入大厅 → 选题 → 进入游戏页对话。
2. AI 在单局内对玩家问题 **仅输出**「是」「不是」「无关」（允许附加极短格式说明，见 6.3，默认关闭）。
3. 用户可 **查看完整汤底** 并看到自己的 **提问历史（推理过程）**；也可通过 **提交真相** 在猜中核心要素后 **胜利揭晓** 并进入汤底页。
4. 单局流程可 **正常结束**（提交真相胜利 / 查看汤底 / 结束游戏），无阻塞性 bug。

### 2.2 成功指标（上线后 4 周，可调整）

| 指标 | 说明 | 目标（建议值） |
|------|------|----------------|
| 开局率 | 进入大厅后点击「开始游戏」占比 | ≥ 40% |
| 破局率 | 在 **未使用「查看汤底」** 的前提下，通过「提交真相 / 我猜到了」被 AI 判定为 **猜中核心要素**（胜利）而进入汤底页的会话，占 **已开始游戏会话** 的比例 | ≥ 8%–15%（首版题库与判定难度不同，可季度复盘调整） |
| 放弃率 | 用户点击「结束游戏」并确认，且 **全程未进入汤底页**（既未胜利揭晓、也未「查看汤底」）的会话，占 **已开始游戏会话** 的比例 | ≤ 35%–45%（过高说明挫败或引导不足） |
| 揭晓率 | 用户点击「查看汤底」并确认后进入汤底页的会话，占 **已开始游戏会话** 的比例 | ≥ 20%–30%（与破局率共同反映「愿意看到结局」的比例） |
| 次留 | 次日回访 | ≥ 15%（首版可放宽） |
| 技术 | 问答接口 P95 延迟 | < 3s |

### 2.3 非目标（MVP 不做）

- 用户注册登录、云端存档、排行榜、多人房间、用户自制题目、付费体系。

## 3. 目标用户

| 用户群 | 特征 | 核心诉求 |
|--------|------|----------|
| 推理爱好者 | 玩过海龟汤或类似推理 | 题目有意思、回答不耍赖、愿意复玩 |
| 上班族 / 碎片用户 | 时间碎片化 | 上手快、单局 5–15 分钟、操作少 |
| AI / 技术爱好者 | 关注 AI 应用 | 对话自然、规则清晰、可观察「主持」边界 |

**典型场景**

- 睡前或通勤：开一局轻松难度。
- 朋友分享链接：点开即玩，无需安装。

## 4. 名词与规则

| 名词 | 定义 |
|------|------|
| 汤面 | 展示给玩家的不完整故事 / 谜面 |
| 汤底 | 完整真相与关键因果，开局对玩家隐藏 |
| 主持人 / AI | 根据汤底判定玩家问题，只答「是 /不是 / 无关」 |
| 一局（Session） | 从选题进入游戏页到结束游戏或进入汤底页的一次游玩过程 |

**玩法规则（产品层）**

1. 玩家用 **自然语言** 提问（建议引导为可判断真假的陈述或 Yes/No 问题）。
2. AI **不得** 主动剧透汤底；**不得** 在回答中复述完整汤底（除非进入汤底页或产品明确允许的「揭晓」流程）。
3. 三类回答语义约定：
   - **是**：与汤底一致、问题指向为真。
   - **不是**：与汤底矛盾、问题指向为假。
   - **无关**：无法据此在汤底框架下判断真假，或与推理推进无关键关联（由 AI 依据内置汤底判定）。

## 5. 信息架构与页面地图

```
首页 / 游戏大厅
    └── 游戏页（对话）
            ├── [提交真相·猜中] → 汤底页（胜利反馈）
            ├── [查看汤底] → 汤底页
            └── [结束游戏] → 确认 → 返回大厅 或 汤底页（可选，见 7.2）
汤底页
    └── [再来一局] → 游戏大厅
```

## 6. 核心功能

### 6.1 MVP（必须做）

#### 6.1.1 游戏大厅

**用户故事**

- 作为玩家，我希望看到 **可选海龟汤列表**，以便按兴趣开始游戏。
- 作为玩家，我希望看到 **难度与类型**，以便选择合适的一局。

**功能说明**

| 编号 | 功能点 | 说明 |
|------|--------|------|
| H-1 | 游戏列表 | 展示题目卡片列表；每条至少包含：标题、难度标签、类型标签（若有）、简短导语（可选，1 行） |
| H-2 | 筛选 / 排序（可选，建议 MVP 简化为「全部」） | 至少支持按难度筛选；排序默认「推荐」或「难度从低到高」 |
| H-3 | 开始游戏 | 点击卡片进入 **游戏页**，并加载对应 `storyId` 的汤面 |
| H-4 | 空状态 | 列表为空时提示「暂无题目」及刷新 |

**数据依赖**

- 题目元数据来自静态配置或后端接口（见 11.2）。

---

#### 6.1.2 游戏页面

**用户故事**

- 作为玩家，我希望看到 **汤面**，并在下方 **像聊天一样提问**。
- 作为玩家，我希望 **只看到「是 / 不是 / 无关」**，以保持海龟汤体验。
- 作为玩家，我希望 **看到完整对话历史**，方便回顾。
- 作为玩家，我可以通过 **「提交真相 / 我猜到了」** 提交结论，由 AI 判定是否猜中核心要素。
- 作为玩家，我可以通过 **「查看汤底」** 提前揭晓，或通过 **「结束游戏」** 放弃本局。
- 作为玩家，若对某条「是 / 否 / 无关」回答存疑，可发起 **重新判定**，让 AI 结合上下文再判一次。

**功能说明**

| 编号 | 功能点 | 说明 |
|------|--------|------|
| G-1 | 汤面展示 | 页面顶部或独立区域展示当前题 **汤面**全文；长文可折叠「展开」 |
| G-2 | 聊天区 | 用户消息右对齐（或统一会话样式）；AI 消息左对齐；时间戳可选 |
| G-3 | 输入与发送 | 多行文本框 + 发送按钮；支持 Enter 发送（Shift+Enter 换行）；空内容不可发送 |
| G-4 | AI 回答格式 | **默认仅展示**「是」「不是」「无关」三选一；不展示长篇解释（见 6.3） |
| G-5 | 对话历史 | 当前局内所有问答按时间顺序展示；进入页面后拉取或本地恢复本会话记录 |
| G-6 | 查看汤底 | 按钮点击后 **二次确认**（防误触）：「确定要查看汤底吗？将无法继续本局推理。」确认后跳转 **汤底页** |
| G-7 | 结束游戏 | 结束本局：清除或标记会话为已结束；返回 **游戏大厅**（或可选跳转汤底页，产品可配置，默认回大厅） |
| G-8 | Loading | 等待 AI 回答时展示加载态，禁用重复发送或排队发送 |
| G-9 | 异常 | 网络失败时提示重试；连续失败可提供「复制错误信息」便于反馈 |
| G-10 | 提交真相 / 我猜到了 | 独立入口（如次要按钮或底部操作区）：玩家输入 **一整段结论**（非单轮「是/不是」问答）并提交；后端将结论与 `truth` + **`core_conditions`（见 11.1）** 一并交给 AI 判定。**猜中核心要素**：展示 **胜利反馈**（文案 + 可选动效），会话标记为胜利揭晓，跳转 **汤底页** 展示完整汤底与本局对话。**未猜中**：在同会话内插入系统提示（如「不完全对，继续努力」），**不**揭晓汤底，玩家可继续提问或再次提交。**防爆破限制**：单局最多允许 **提交 3 次**（含未猜中的提交），前后端均需校验；界面需展示 **剩余次数**（如「还有 2 次机会」）。次数耗尽后，「提交真相 / 我猜到了」**置灰或隐藏**，并固定提示：「提交次数已用完，请继续提问或直接查看汤底」，引导玩家回到常规问答或「查看汤底」路径 |
| G-11 | 异常纠偏 / 重新判定 | 在 **每条 AI「是 / 不是/ 无关」消息**上提供「重新判定」操作（如次要文字按钮或菜单项）：点击后将 **该条用户问题 + AI 原回答 + 当前对话上下文** 发送至后端，由大模型 **结合上下文重新输出** 一次「是 / 不是 / 无关」（或等价枚举）。**替换规则**：以新回答覆盖该条 AI 消息展示，并在会话中保留「已重新判定」标记（避免无限刷判，可限每局 N 次，建议默认 3 次，可配置） |

**边界**

- 「提交真相」与常规 **单轮提问** 在交互上需区分（避免把结论误发成普通提问）；可在产品层约定：G-10 使用独立表单或独立按钮触发的模态，与 G-3 普通发送分流。
- 「重新判定」仅针对 **是/不是/无关类回答**，不改变汤底与 `core_conditions`；用于缓解模型偶发不一致，**不是**修改题目规则。

---

#### 6.1.3 汤底页面

**用户故事**

- 作为玩家，我希望看到 **完整汤底**，满足好奇心。
- 作为玩家，我希望看到 **自己的推理过程（提问与 AI 回答）**，便于复盘。
- 作为玩家，我希望 **再来一局** 快速回到大厅选题。

**功能说明**

| 编号 | 功能点 | 说明 |
|------|--------|------|
| T-1 | 完整汤底 | 展示标题 + 汤底正文；可分段（背景 / 关键转折） |
| T-2 | 推理过程 | 复用游戏页对话列表（只读）；与汤底区上下分区或 Tab |
| T-3 | 再来一局 | 跳转 **游戏大厅**；本局会话可关闭 |
| T-4 | 返回 | 可选：返回首页；浏览器返回行为不导致死循环 |

---

### 6.2 AI 行为与工程约束（MVP 必守）

#### 6.3 回答规范

| 项目 | 要求 |
|------|------|
| 输出 | 模型输出应被后端解析为枚举：`YES` / `NO` / `IRRELEVANT` |
| 禁剧透 | 不在游戏页返回汤底原文或等价复述 |
| 一致性 | 同一事实被不同问法指向时，结论应一致（依赖 Prompt + 可选结构化汤底，见 11.3） |

**可选（默认关）**

- 「简短理由」一行：仅用于调试或高级模式，**MVP 默认关闭**，避免破坏纯净体验。

#### 6.4 输入处理

- 空输入、纯空格：前端拦截。
- 过长输入：前端限制字符数（建议 500 字）+ 后端截断或拒识并提示。
- 敏感内容：走内容安全策略（见 9.3）。

## 7. 用户流程（文字版）

### 7.1 主流程：从进入到揭晓

1. 用户打开网站 → **游戏大厅**。
2. 点击某题「开始游戏」→ **游戏页**，加载汤面。
3. 用户输入问题 → 请求后端 → AI 返回「是/不是/无关」→ 写入对话历史；若对某条回答存疑，可对 **该条 AI 回答** 点击「重新判定」→ 后端结合上下文再判一次 → 更新该条展示（见 G-11）。
4. 用户点击「提交真相 / 我猜到了」→ 输入结论并提交 → 后端依据汤底与 **`core_conditions`** 判定：
   - **猜中核心要素** → 展示 **胜利反馈** → 进入 **汤底页**（揭晓汤底 + 本局对话，会话为胜利结束）；
   - **未猜中** → 提示「不完全对，继续努力」→ 留在 **游戏页**，可继续提问或再次提交。
5. 用户也可不提交真相，直接点击「查看汤底」→ 确认 → **汤底页**，展示汤底 + 本局对话（非胜利路径，统计上计入「揭晓率」见 2.2）。
6. 用户点击「再来一局」→ **游戏大厅**。

### 7.2 结束游戏（未揭晓）

1. 用户点击「结束游戏」→ 二次确认「确定结束本局？」。
2. 确认后：**默认** 返回游戏大厅，本会话标记结束；**不**强制展示汤底（与「查看汤底」区分）。
3. 若产品希望「结束也进汤底页」，可在实现前改 PRD 单条款，需统一按钮文案避免混淆。

## 8. 界面设计要求

### 8.1 整体风格

- **调性**：神秘、悬疑、沉静。
- **主色**：**深蓝色**系背景（如 `#0f172a` ~ `#1e3a5f` 区间，具体以设计稿为准）。
- **强调色**：低饱和金 / 青 / 紫其一，用于按钮与关键信息，避免刺眼荧光。
- **字体**：中文优先无衬线，保证可读性；标题可略具装饰感。
- **动效**： subtle（微动效），揭晓汤底时可略增强仪式感（如淡入、分隔线动画）。

### 8.2 游戏大厅

- **布局**：**卡片式**网格（响应式：移动端单列，桌面多列）。
- **卡片内容**：故事标题、难度标签、类型标签、可选一句导语。
- **交互**：悬停 / 点击态清晰；主按钮「开始游戏」在卡片内或卡片底部。

### 8.3 游戏页面

- **风格**：**类即时通讯**（参考微信会话）：上方信息区（汤面），下方消息流 + 底部输入框。
- **消息气泡**：用户与 AI 区分明显；AI 回答仅显示「是 / 否 / 无关」时可用标签样式强化。
- **固定底栏**：输入框与「提交真相 / 我猜到了」「查看汤底」「结束游戏」等操作区不遮挡最后一条消息（需预留安全区）；「重新判定」挂在对应 AI 气泡上，不占用主底栏。

### 8.4 汤底页面

- **仪式感**：汤底区域与「推理过程」区视觉分层；汤底可用卡片 / 卷轴 / 幕布揭开等隐喻（实现成本可控前提下）。
- **推理过程**：时间线或聊天只读列表，与游戏页一致更易理解。

### 8.5 无障碍与基础体验

- 对比度符合 WCAG AA 为努力方向（深色背景下注意文字对比）。
- 关键操作可键盘聚焦；按钮有明确 `aria-label`（后续迭代）。

## 9. 非功能需求

### 9.1 性能

- 首屏可交互：目标 **3s 内**（弱网可放宽，需骨架屏）。
- 单次问答：**P95 < 3s**（含模型与网络）。

### 9.2 兼容

- 浏览器：最新两版 Chrome / Edge / Safari；移动端 Safari / Chrome。
- 响应式：**移动优先**，桌面增强。

### 9.3 安全与合规

- HTTPS。
- 用户输入与 AI 输出建议过 **内容安全**（政治、色情、暴力等策略按国内合规要求配置）。
- API Key **仅服务端**，禁止前端暴露。

### 9.4 可观测性

- 前端：关键事件埋点（进入大厅、开始游戏、发送问题、**提交真相**（含胜负结果）、**重新判定**（含是否变更结果）、查看汤底、结束游戏、错误）。
- 后端：请求日志、错误率、延迟；**不**记录完整用户隐私字段于明文日志。

## 11. 数据与接口（草案）

### 11.1 核心实体

**Story（题目）**

```ts
interface Story {
  id: string;
  title: string;
  difficulty: 'easy' | 'medium' | 'hard';
  category: string;        // 如：悬疑、日常、恐怖
  surface: string;         // 汤面
  truth: string;           // 汤底全文（仅服务端完整存储，前端仅在汤底页展示）
  core_conditions: string[]; // 核心判定条件：将汤底拆解为 3–5 条关键事实，供 AI 锚定判定（不单独对玩家展示，见 11.3）
}
```

**Session（一局）**

```ts
interface Session {
  sessionId: string;
  storyId: string;
  messages: Array<{ role: 'user' | 'assistant'; content: string; createdAt: string }>;
  status: 'playing' | 'revealed' | 'aborted';
}
```

### 11.2 API 草案（REST 示例）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/stories` | 题目列表（不含 truth） |
| GET | `/api/stories/:id` | 题目详情（游戏页：仅 surface；或分字段控制） |
| POST | `/api/sessions` | 创建会话 `storyId` |
| POST | `/api/sessions/:id/messages` | 用户提问 body: `{ text }` → 返回 `{ answer: 'YES'|'NO'|'IRRELEVANT' }` |
| POST | `/api/sessions/:id/guess` | 提交真相 body: `{ text }` → 返回 `{ win: boolean }` 等；`win: true` 时前端跳转汤底页并展示胜利反馈（判定依赖 `truth` + `core_conditions`，见 11.3） |
| POST | `/api/sessions/:id/messages/:assistantMessageId/rejudge` | 对某条 AI 回答重新判定 → 返回 `{ answer: 'YES'|'NO'|'IRRELEVANT' }`（见 G-11） |
| GET | `/api/sessions/:id` | 获取会话与消息（恢复页面） |
| POST | `/api/sessions/:id/reveal` | 标记揭晓（可选） |
| DELETE | `/api/sessions/:id` | 结束并清理（可选） |

**说明**：`truth` 仅在 **汤底接口** 或 **揭晓后专用接口** 返回，避免误配导致泄漏；**`core_conditions` 仅服务端存储与参与 AI 判定**，不下发至前端（与 11.1 注释一致）。**「提交真相」** 接口须做 **单局次数上限** 校验（默认 **3 次**，与 G-10 一致），防止恶意刷接口套答案。

### 11.3 AI 实现建议（研发参考）

1. **System Prompt** 固定：角色为海龟汤主持人，只输出三类答案（或「提交真相」接口要求的结构化胜负结果）。
2. **User 消息**：附带「汤面 + 汤底」在 **服务端** 组装，**不**发给前端。
3. **结构化事实锚点（强依赖）**：服务端在每次调用 AI 做 **是非判定**（回答玩家「是/否/无关」）以及 **胜负判定**（「提交真相」是否猜中）时，除传入 `truth` 全文外，**必须同时传入 `core_conditions`**。模型应 **优先以 `core_conditions` 为锚** 进行推理与对齐：  
   - 将玩家陈述、玩家问题与 **各条核心条件** 比对，减少游离于全文叙述之外的幻觉与前后不一致；  
   - 「猜中」的语义定义为：玩家结论覆盖或等价于 **核心条件所表达的关键因果/事实**（具体匹配策略在实现中可约定为「全部命中」或「关键子集命中」，需在评审时写死并写入 Prompt）。  
4. **上下文截断与记忆策略**：为控制 **Token 成本** 并保证 **响应速度**，建议采用 **滑动窗口**：在 **常规问答**（玩家逐轮提问、AI 答「是/否/无关」）中，每次请求仅携带 **汤面**、**`core_conditions`** 以及 **最近 5–8 轮** 对话（用户问 + AI 答计为一轮，具体上限可在实现中微调）。**仅在** 触发 **「重新判定」**（G-11）或 **「提交真相」**（G-10）时，再 **视情况拉长上下文窗口**（例如纳入更早的关键轮次或完整会话），以便纠偏与胜负判定更稳，但仍需设 **最大长度** 防止超时与爆账。  
5. **与 6.1.2 的衔接**：`core_conditions` 也是 G-10「提交真相」判定的主要依据；常规问答（G-4）同样应受 `core_conditions` 约束，以保证与胜利条件口径一致。  
6. 后续优化：在 `core_conditions` 之上可增加程序级校验（如关键词覆盖度打分），与模型输出交叉验证（非 MVP 必做）。

## 12. 验收标准（MVP）

| 编号 | 场景 | 验收条件 |
|------|------|----------|
| AC-1 | 大厅 | 至少展示 **3** 道以上题目，含难度与类型 |
| AC-2 | 游戏 | 发送问题后，界面仅出现 **是/否/无关** 之一（无长篇解释，默认配置） |
| AC-3 | 历史 | 刷新页面后，同一会话（同 session）对话可恢复 **或** 明确产品决策为不恢复并写清提示 |
| AC-4 | 汤底 | 「查看汤底」后展示与题库一致的完整汤底 + 本局问答列表 |
| AC-5 | 结束 | 「结束游戏」后回到大厅，无卡死；会话状态合理 |
| AC-6 | 异常 | 断网有明确提示与重试 |
| AC-7 | 提交真相 | 提交结论后，**猜中**时出现胜利反馈并进入汤底页；**未猜中**时提示「不完全对，继续努力」且不剧透汤底；并验证单局 **3 次** 提交机会耗尽后的 **拦截与提示**（置灰/隐藏 + 文案）是否生效 |
| AC-8 | 重新判定 | 点击「重新判定」后，该条 AI 回答被更新为新结果；达到次数上限后给出友好提示 |

## 14. 风险与依赖

| 风险 | 缓解 |
|------|------|
| 模型偶发输出违规或长篇解释 | 后端解析失败则重试或 fallback；强约束 JSON 输出 |
